# Выявление и устранение "горячих" шардов

## Метрики для мониторинга горячих шардов

1. **Количество операций по шардом (QPS per Shard):** Отслеживание общего количества операций (чтение, запись, обновление) на каждом шарде, чтобы выявить дисбаланс нагрузки между шардом и остальными.
2. **Использование CPU и памяти:** Высокая загрузка CPU и памяти на конкретном шарде — один из ключевых индикаторов перегрева.
3. **Дисковый ввод-вывод (Disk I/O) и сетевой трафик (Network I/O):** Пиковые значения или продолжительная нагрузка на эти параметры могут свидетельствовать о перегруженности шарда.
4. **Время выполнения запросов и латентность:** Среднее время выполнения запросов (read and write latency) помогает понять, замедляется ли шардинг из-за высокой нагрузки на конкретный шард.
5. **Размер и распределение чанков:** Дисбаланс в размере чанков (chunk size) на шарде может привести к тому, что слишком много данных и запросов скапливаются на одном шарде. Важно следить за балансом чанков и использовать balancer для перераспределения данных.
6. **Количество "горячих" чанков (Hot Chunks):** Метрика, указывающая на активность и концентрацию запросов на определенных чанках ("горячие точки").

## Диагностика "горячего" шарда

* `mongostat` - статистика по операциям, подключениями, ресурсам (CPU, память), очередям и др. Можно запустить с подключением к mongos, чтобы видеть данные по всем шардом кластера.
* `mongotop` — время, затрачиваемое на операции чтения и записи по коллекциям, можно контролировать активность и нагрузку на шард.
* `db.serverStatus()` — полный отчёт по многим метрикам, включая операции, память, подключения, задержки, ошибки и пр. (большинство внешних мониторингов также используют ее для сбора данных.)
* `db.collection.stats()` и `db.stats()` — для получения статистики хранения, использования диска, размера данных в коллекциях и базе в целом.
* `analyzeShardKey` — для оценки эффективности shard key путем сбора статистики по выборкам и распределению.
* `balancerStatus` (`config.sh.getBalancerStatus()`) — для мониторинга состояния и активности балансировщика шардинга, миграций чанков между шардом.

## Механизмы перераспределения данных

Автоматическое перераспределение данных в MongoDB основано на балансе чанков с помощью автоматического балансировщика и миграций чанков, что обеспечивает равномерное распределение нагрузки и данных без остановки работы кластера.

Для управления балансировщиком и миграциями чанков через Mongosh можно использовать нижеприведенный код:

```javascript
// 1. Проверяем статус балансировщика
sh.getBalancerState());  // true — включён, false — выключен
sh.status());

// 2. Включение балансировщика (если выключен)
sh.startBalancer();

// 3. Остановка балансировщика (если нужно)
sh.stopBalancer();

// 4. Задание временного окна для работы балансировщика
// Например, запускать балансировщик только с 2:00 до 6:00 утра
db.settings.update(
  { _id: "balancer" },
  { $set: { activeWindow: { start: "02:00", stop: "06:00" } } },
  { upsert: true }
);

// 5. Ручная миграция чанка (пример) 
sh.moveChunk(
  "mydb.mycollection",
  { _id: ObjectId("...") }, // критерий для поиска чанка
  "shard0001" // имя шарда, куда мигрировать
);

// 6. Проверка текущих миграций и статистики
db.currentOp({ "command.moveChunk": { $exists: true } });
```
