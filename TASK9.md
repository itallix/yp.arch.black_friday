# Чтение с реплик и консистентность: products, orders, carts

## Таблица с операциями чтения и записи

Коллекция |	Операция чтения	| Primary | Secondary |	Допустимая задержка репликации | Обоснование
----------|-----------------|---------|-----------|--------------------------------|------------
`orders`| Быстрое создание заказов + списание остатков | Да | Нет |	- | Списывание остатков и создание заказа - критически важные операции, требуют наиболее свежих данных и согласованности, чтобы избежать ошибок и расхождений.
`orders`| Поиск истории заказов конкретного пользователя | Нет | Да | До 1-2 сек | История заказов - типичное чтение с возможной лёгкой задержкой, не влияет критично на логику бизнеса, улучшает масштабируемость.
`orders`| Отображение статуса заказа | Да | Нет | - | Статус заказа - важная информация для пользователя, устаревшая информация приводит к ошибкам в коммуникации и процессах.
`products` | Частые обновления остатков при покупках | Да | Нет | - | Актуальность остатков для предотвращения продажи отсутствующих товаров; критично для бизнес-логики и предотвращения oversell.
`products` | Поиск товаров по категориям и фильтрация | Нет | Да | До 2 сек | Поиск и фильтрация - операции чтения, для которых допустима небольшая задержка ради производительности и масштабируемости.
`products` | Отображение описания товара на странице | Нет | Да | До 2 сек | Описание товара меняется редко, можно использовать данные с небольшой задержкой.
`carts` | Создание корзины / получение текущей корзины по {session_id, status:"active"} или {user_id, status:"active"} | Да | Нет | - | Актуальность текущей корзины важна для пользователя, чтение с primary гарантирует свежесть данных.
`carts` | Добавление или замена товара в корзине | Да | Нет | - | Записи и обновления должны отражаться сразу, чтобы избежать конфликтов и рассогласований.
`carts` | Удаление товара из корзины | Да | Нет | -	| Требуется консистентность для корректного отображения корзины.
`carts` | Слияние гостевой корзины в пользовательскую | Да | Нет | - | Сложная транзакционная логика, требующая актуальных данных и последовательности операций.
`carts` | Отметка корзины как заказанной | Да | Нет | - | Закрытие корзины - важное событие, требует гарантированного применения без задержек.

## Обоснование выбора

- Для всех операций, изменяющих состояние данных или критичных для бизнес-логики (создание заказов, списание остатков, обновление корзины, статус заказа) чтение и запись должны идти с primary, чтобы обеспечить максимально свежие и согласованные данные.
- Для операций, связанных с историей и аналитикой (поиск истории заказов, поиск товаров, чтение описания) можно использовать secondary с небольшой задержкой репликации (1–2 секунды), что улучшает масштабируемость без значительной потери качества данных.
- Задержка репликации в пределах 1-2 секунд приемлема для большинства пользовательских интерфейсов при чтении с secondary, учитывая скорость распространения изменений в MongoDB репликах.
- При слиянии корзин и обновлении текущей корзины необходимо чтение с primary, чтобы избежать конфликтов и обеспечить корректность консолидированных данных.
